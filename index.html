When Reads Became the Bottleneck
Abdelkader Bouadjadja Â· February 2026 Â· 8 min read

Between 2020 and 2025, I was part of the engineering leadership at Hungerstation ~ at the time one of the largest food and grocery delivery platforms in Saudi Arabia, operating across dozens of cities with millions of active users. When I joined, the platform was already at significant scale. By the time I left, order volume had grown roughly 8Ã—.

That kind of growth doesn't happen in a vacuum. It compresses every latent weakness in your architecture into something urgent. Queries that were slightly slow become completely unacceptable. Patterns that were fine at 1Ã— traffic become existential at 8Ã—. The system stops rewarding optimism and starts demanding honesty.

One of the most persistent and consequential problems we faced was deceptively simple to describe: we were reading too much, too often, from the wrong place.

At a platform serving two very different product worlds ~ food delivery and grocery ~ almost every user interaction fans out into multiple read paths before anything meaningful happens. Search results, category pages, availability checks, pricing rules, eligibility constraints, campaign logic, compliance rules. Writes matter, but reads are where the system lives or dies.

For a long time, the architecture held up. Then it didn't.

The First Signs of Tension

Nothing broke overnight. There was no single incident, no dramatic outage, no obvious failure that forced an immediate rewrite. Instead, we started seeing patterns that made us uncomfortable:

  • Latency percentiles creeping up during peak traffic
  • Read replicas becoming more sensitive to bursts
  • Small changes in load producing disproportionate impact

Individually, none of these were alarming. Together, they pointed to a deeper issue: our read path was becoming structurally coupled to database pressure. The usual playbook still works at this stage ~ add replicas, tune queries, cache responses, buy time. But buying time is not the same as buying resiliency.

Stepping Back: What Are We Really Doing?

We paused and forced ourselves to restate the problem in plain terms. Most of our read-heavy flows depend on data that:

  • Changes infrequently compared to request volume
  • Is highly structured
  • Is filtered and combined in predictable ways

Yet we were re-reading that data from disk, repeatedly, under load, for every request. That raised a simple but uncomfortable question:

Why are we relying on database reads in the hot path when the data itself is largely stable and deterministic?

That question pushed us away from incremental optimizations and toward a more fundamental rethink.

The Decision: Treat Products as In-Memory State

The idea we landed on was deliberately bold:

  • Every pod loads the full active product dataset at startup
  • Data lives in memory ~ not as a cache, but as authoritative state
  • Reads become pure computation
  • Database access moves out of the critical path

This was not a decision we took lightly. But it offered something compelling: predictability.

How This Works in a Kubernetes Environment

Each pod follows a strict lifecycle:

  • Startup: full synchronization before serving any traffic ~ build in-memory structures, validate consistency, only then mark ready. No partial readiness, no warm-up while serving.
  • Running: updates applied incrementally. Product changes streamed as events, applied atomically in memory. No full reloads, no blocking operations.
  • Failure: pods remain disposable. If one dies, it restarts and rebuilds state deterministically. Resiliency comes from clean rebuild, not preserved state.

Filtering Without the Database: Roaring Bitmaps

Once data lives in memory, the problem shifts from querying to computing. NaÃ¯ve in-memory filtering doesn't scale ~ iterating over thousands of products per request with branching logic quickly becomes CPU-heavy and unpredictable.

This is where Roaring Bitmaps became central to the design. A Roaring Bitmap is a compressed data structure for representing large sets of integers efficiently. More importantly, it enables extremely fast set operations: AND, OR, NOT.

Each product is represented by an integer ID. Each filter dimension ~ availability, category, eligibility, flow constraints ~ is a bitmap. At request time, filtering becomes bitmap intersections. The CPU does the work, in a way that is highly optimized and cache-friendly: 

// Precomputed structures
active   := roaring.Bitmap
byCity   := map[int]*roaring.Bitmap
byCategory := map[int]*roaring.Bitmap

func eligibleProducts(cityID, categoryID int) []int {
    result := roaring.And(
        active,
        byCity[cityID],
        byCategory[categoryID],
    )
    return result.ToArray()
}

Reads become deterministic, in-memory computation. Latency becomes a function of CPU, not I/O contention.

What Changed for Resiliency

The most immediate impact was on system behavior under load. Traffic spikes no longer translated into database stress. The read path scaled with CPU. The database didn't disappear ~ but it stopped being the first thing to suffer when traffic increased. That alone significantly reduced blast radius.

Perhaps more importantly, the system became easier to reason about. We could model worst-case behavior, test rebuild scenarios, and understand failure modes with much more clarity than before.

Tradeoffs We Accepted Deliberately

This approach is not free:

  • Memory usage per pod is higher
  • Startup logic is more complex
  • Correctness of synchronization mechanisms becomes critical
  • New classes of bugs become possible

We believe there is no perfect solution in engineering. Every architecture shifts complexity somewhere. Moving complexity into memory management and rebuild logic was a better fit for us than continuing to push the database harder. We chose this balance because it aligned with our traffic patterns, our product constraints, and our ability to reason about the system as it grows.

How a Small Team Made This Work

This wasn't about heroics or brute force. We kept scope intentionally small at first, focusing only on the read paths that mattered most. We rolled out progressively, observed under real traffic, and expanded carefully.

We spent a disproportionate amount of time defining invariants: what must always be true, what happens on restarts, what "ready" really means. Those constraints shaped the design more than any single optimization. And we were honest about failure modes ~ we assumed things would go wrong and designed for recovery, not perfection.

That discipline ~ more than any specific technology ~ is what allowed a team of a few engineers to ship this safely.

Closing Thought

Scaling databases is often the first instinct. Sometimes, the better question is whether the database should be in the critical path at all.

Resiliency isn't a milestone you reach. It's something you keep earning as the system evolves.
